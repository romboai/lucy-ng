# Roadmap: lucy-ng v6.0 Skill Quality Overhaul

## Overview

v6.0 is a targeted quality overhaul of all skill and agent definitions. No new Python code — every phase edits .md files in ~/.claude/commands/lucy-ng/ and ~/.claude/agents/. The work divides naturally into four layers: skill architecture (restructuring case.md and archiving legacy artifacts), agent intelligence (4J coupling awareness and message validation), skill UX (triggering, routing, error recovery), and operational infrastructure (version checking and smoke testing).

## Phases

- [ ] **Phase 55: Skill Architecture** - Factor case.md, archive legacy agent, extract shared NMR reference tables
- [ ] **Phase 56: Agent Intelligence** - Add 4J coupling awareness across nmr-chemist/lsd-engineer/solution-analyst and orchestrator message validation
- [ ] **Phase 57: Skill UX** - Optimize all skill descriptions, add routing decision tree, sanitise dry-run, predict and dereplicate error recovery
- [ ] **Phase 58: Operations** - Add version compatibility check to status skill and smoke test mode to case skill

## Phase Details

### Phase 55: Skill Architecture
**Goal**: Users and agents work with a lean, non-duplicated skill set — case.md fits in context, the legacy agent is clearly retired, and shared reference tables live in one place
**Depends on**: Nothing (first phase of milestone)
**Requirements**: ARCH-01, ARCH-02, ARCH-03
**Success Criteria** (what must be TRUE):
  1. case.md is under 500 lines with explicit read-file references to progress-format.md, loop-patterns.md, and advisory-templates.md
  2. lucy-case-agent.md has a deprecation header stating it is retired and not spawned by any active workflow
  3. A references/nmr-basics.md file exists with shared experiment types and chemical shift regions
  4. Each active agent that previously duplicated NMR reference tables references nmr-basics.md instead
**Plans**: 2 plans
Plans:
- [x] 55-01-PLAN.md — Factor case.md into core flow plus 3 extracted reference files
- [x] 55-02-PLAN.md — Archive legacy agent, create shared NMR reference, update nmr-chemist

### Phase 56: Agent Intelligence
**Goal**: The CASE team correctly identifies, defers, and verifies 4J HMBC couplings, and the orchestrator rejects malformed inter-agent messages rather than silently proceeding with partial data
**Depends on**: Phase 55
**Requirements**: INTL-01, INTL-02, INTL-03, INTL-04
**Success Criteria** (what must be TRUE):
  1. nmr-chemist lists potential 4J correlations as a distinct category in [SETUP-COMPLETE] messages when 4+ aromatic carbons are detected (110-160 ppm)
  2. lsd-engineer defers flagged 4J correlations to later HMBC batches and skips them when solutions already exist
  3. solution-analyst runs lucy predict c13 on top candidates to confirm aromatic carbon presence rather than relying solely on the warnings array
  4. Orchestrator checks each structured message ([SETUP-COMPLETE], [ITERATION-COMPLETE], [RANKING-COMPLETE]) for required fields and requests resend when fields are missing
**Plans**: 2 plans
Plans:
- [ ] 56-01-PLAN.md — Add 4J flagging to nmr-chemist and message validation to orchestrator
- [ ] 56-02-PLAN.md — Add 4J deferral to lsd-engineer and aromatic verification to solution-analyst

### Phase 57: Skill UX
**Goal**: Users reaching lucy-ng skills through natural language get routed correctly, skill descriptions match how users actually ask, and error states in predict/dereplicate/sanitise give actionable next steps
**Depends on**: Phase 55
**Requirements**: SKUX-01, SKUX-02, SKUX-03, SKUX-04, SKUX-05
**Success Criteria** (what must be TRUE):
  1. All 5 skill descriptions include natural-language trigger phrases (NMR, structure determination, unknown compound, blind testing, chemical shifts)
  2. The routing page (lucy-ng.md) contains a decision tree with at least 4 branches guiding users to the correct sub-command
  3. sanitise.md describes a dry-run step that reports findings without modifying files and requires explicit user confirmation before proceeding
  4. predict.md provides specific guidance when HOSE code lookup returns no predictions (alternative SMILES, database coverage note)
  5. dereplicate.md provides specific guidance when 0 matches are returned (formula spelling, related formulas, compound not in database)
**Plans**: 2 plans
Plans:
- [ ] 57-01-PLAN.md — Add trigger phrases to all 5 skill descriptions, build routing decision tree
- [ ] 57-02-PLAN.md — Add error recovery to predict/dereplicate, add dry-run to sanitise

### Phase 58: Operations
**Goal**: The status skill reports CLI version incompatibility before any workflow begins, and /lucy-ng:case can be validated in one iteration without running to convergence
**Depends on**: Phase 55
**Requirements**: OPER-01, OPER-02
**Success Criteria** (what must be TRUE):
  1. status.md checks lucy --version output against a documented minimum required version and prints a clear incompatibility warning when below minimum
  2. A smoke test invocation of /lucy-ng:case runs exactly 1 CASE iteration, verifies that the team spawned, NMR-chemist picked peaks, LSD-engineer built a file, and devils-advocate validated it, then stops and reports pass/fail
  3. Smoke test result is distinguishable from a normal CASE run (explicit SMOKE TEST header in output)
**Plans**: 1 plan
Plans:
- [ ] 58-01-PLAN.md — Add version check to status skill, add smoke test mode to case skill
