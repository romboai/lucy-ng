# lucy-ng Roadmap — v5.0 Fragment Library

## v5.0 Fragment Library (Phases 49-54)

**Milestone Goal:** Build a substructure-subspectrum correlation (SSC) fragment library from the existing 928K-compound database, enabling fragment search and DEFF/FEXP goodlist constraint injection to dramatically reduce LSD solution counts.

**Strategic context:** In Sherlock (Wenk thesis), the 24.5M SSC fragment library reduced 34/40 cases to a single solution. This is the highest-leverage missing capability in lucy-ng today.

**Motivation:** v4.0 team-based CASE fixed all constraint-loss bugs and added aromatic ring awareness, but solution counts remain high because LSD has no knowledge of which substructures are chemically plausible for a given spectrum. Fragment constraints encode prior structural knowledge — every known compound's substructures become searchable constraints.

### Phase overview:

- [ ] **Phase 49: Fragment Schema and Infrastructure** - Schema v7 with ssc/ssc_bitset tables in separate lucy-ng-fragments.db, Pydantic models, DatabaseManager query methods (completed -)
- [ ] **Phase 50: SSC Extraction Pipeline** - BFS sphere fragmentation over 928K compounds with checkpointing, fingerprint generation, bin size validation before full run (completed -)
- [ ] **Phase 51: Fragment Search Engine** - Batched bitset pre-screening + fine spectral matching CLI with JSON output (completed -)
- [ ] **Phase 52: DEFF Formatter and LSD File Generation** - SSTR/LINK fragment file format, validated goodlist syntax, lucy fragment to-lsd CLI (completed -)
- [ ] **Phase 53: Agent Integration** - lsd-engineer fragment search step, sequential injection protocol, constraint inventory DEFF/FEXP tracking (completed -)

### Phase 49: Fragment Schema and Infrastructure
**Goal**: Storage infrastructure exists for SSC data — schema v7 tables, Pydantic models, and DatabaseManager query methods are ready before any extraction code runs
**Depends on**: Phase 48 (v4.0 complete)
**Requirements**: FRAG-01
**Success Criteria** (what must be TRUE):
  1. `lucy-ng-fragments.db` file is created with schema v7 (`ssc` and `ssc_bitset` tables) and `schema_meta` table containing bin_size record
  2. `lucy fragment info` reports schema version, SSC count (0), and database file path — confirms file is accessible
  3. Existing `lucy dereplicate c13` and `lucy predict c13` commands run without error after schema change — separate file does not affect main database
  4. `SSCRecord` and `SSCMatch` Pydantic models can be instantiated and serialized to JSON
  5. `DatabaseManager` query methods (`insert_ssc_batch`, `get_ssc_count`, `iter_ssc_bitsets`, `get_ssc_by_id`) callable from Python without error

### Phase 50: SSC Extraction Pipeline
**Goal**: 928K compounds extracted to ~24M SSCs with checkpointing — pipeline survives interruption, bin size validated before full run, aromatic standardization applied consistently
**Depends on**: Phase 49 (schema exists, write methods available)
**Requirements**: FRAG-02, FRAG-03, FRAG-04
**Success Criteria** (what must be TRUE):
  1. `lucy fragment build --sample 1000` completes on 1K compounds, self-search recall >99% on 100 sampled compounds (bin size validated before full extraction)
  2. `lucy fragment build` (full run) completes and `lucy fragment info` reports SSC count in the millions
  3. Interrupted extraction (Ctrl-C mid-run) resumes from last checkpoint with `lucy fragment build --resume` — no duplicate SSCs inserted
  4. `lucy fragment build --fresh` discards existing data and restarts from zero
  5. Skipped compound count (missing atom-indexed shifts) logged to stderr during build

### Phase 51: Fragment Search Engine
**Goal**: Fragment search CLI delivers ranked fragments for a given shift list in under 2 seconds — pre-screening eliminates non-matching SSCs, fine matching filters by DEV/AVGDEV/multiplicity
**Depends on**: Phase 50 (SSC table populated)
**Requirements**: SRCH-01, SRCH-02, SRCH-03, SRCH-04, SRCH-05, SRCH-06
**Success Criteria** (what must be TRUE):
  1. `lucy fragment search --shifts "155.08,151.58,130.2,..." --format json` returns ranked fragments with `deff_commands` and `fexp_command` fields as exact LSD strings
  2. Fragment results are ranked by heavy atom count descending then AVGDEV ascending — largest structurally informative fragment appears first
  3. Search completes in under 2 seconds for a typical 13C shift list on M1 Mac — batched bitset pre-screening does not load full database into RAM at startup
  4. `lucy fragment info` reports library statistics (SSC count, database file size, bin size)
  5. Boolean AND pre-screening is applied before fine matching — running with `--verbose` shows pre-screen candidate count and fine-match count separately

### Phase 52: DEFF Formatter and LSD File Generation
**Goal**: Fragment SMILES are converted to valid LSD DEFF goodlist files — syntax validated with a working LSD smoke test before any agent depends on it
**Depends on**: Phase 49 (SSCMatch Pydantic model available); can run in parallel with Phase 51
**Requirements**: LINT-01, LINT-02, LINT-03
**Success Criteria** (what must be TRUE):
  1. `lucy fragment to-lsd "SMILES"` generates a fragment `.lsd` file with correct SSTR/LINK atom notation — file passes LSD syntax validation
  2. LSD smoke test: inject a known fragment DEFF/FEXP into a minimal LSD problem and confirm solution count decreases (goodlist semantics verified — not accidentally acting as badlist)
  3. DEFF/FEXP commands appear before MULT commands in any LSD file that includes fragment constraints — LINT-03 ordering verified by smoke test
  4. Fragment file is written to the current working directory with a deterministic filename based on fragment SMILES hash

### Phase 53: Agent Integration
**Goal**: lsd-engineer runs fragment search before each LSD iteration and injects the best fragment as DEFF/FEXP — with sequential validation, constraint inventory tracking, and devils-advocate pre-run file check
**Depends on**: Phase 52 (complete CLI pipeline — search + formatter both validated with LSD smoke test)
**Requirements**: AGNT-01, AGNT-02, AGNT-03, AGNT-04
**Success Criteria** (what must be TRUE):
  1. lsd-engineer runs `lucy fragment search --format json` at the start of every iteration and prepends DEFF/FEXP output before MULT commands in the LSD file
  2. If fragment injection yields zero solutions, lsd-engineer discards that fragment, logs the conflict in CASE-PROGRESS.md, and continues to next iteration without the fragment — workflow does not halt
  3. Fragment constraints appear in the constraint inventory JSON block under a `deff_fexp` section — constraint inventory diff by devils-advocate detects fragment addition/removal
  4. Devils-advocate verifies the fragment `.lsd` file exists on disk before approving each LSD solver run
  5. CASE-PROGRESS.md shows fragment search results per iteration (fragment SMILES, heavy atom count, AVGDEV) so the team can review what was applied
**Plans**: TBD

### Phase 54: Multi-Compound UAT
**Goal**: Fragment library reduces solution counts across 5+ test compounds — self-search recall confirmed >99%, real CASE runs show measurable solution count reduction
**Depends on**: Phase 53 (full agent pipeline operational)
**Requirements**: VALD-01, VALD-02
**Success Criteria** (what must be TRUE):
  1. Self-search validation: 100 compounds' own spectra find their own SSCs with >99% recall — confirms fingerprint indexing is correct
  2. 5+ compounds from Sherlock test set run with full CASE pipeline; solution counts before and after fragment injection are recorded and compared
  3. At least 3 of 5 test compounds show a reduction in final solution count when fragment injection is active
  4. Fragment conflict rate (fragments discarded due to zero solutions) is logged per compound — identifies pathological cases for v5.1 investigation
  5. UAT report identifies which remaining solution count failures are attributable to 4J HMBC exclusion (v4.0 known issue) vs. fragment library gaps
