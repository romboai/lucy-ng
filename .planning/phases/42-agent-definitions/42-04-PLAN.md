   - Job: Validate every LSD file before solver runs, diff constraints between iterations, catch dropped constraints, flag issues with severity classification

3. `<domain_knowledge>` section (~130-180 lines):

   **Constraint Diff Protocol (~40 lines — NEW content):**

   For iteration N > 1, compare current LSD file against previous iteration's file:

   a. **Count comparison:** Count each constraint type in both files:
      - MULT definitions (should be identical across iterations)
      - HSQC correlations (should be identical)
      - HMBC correlations (should increase or stay same, never decrease unintentionally)
      - BOND constraints (should persist unless explicitly removed with reasoning)
      - LIST/ELEM/PROP constraints (should persist)
      - DEFF NOT patterns (should ALWAYS persist — dropping these was v3.0 bug #1)
      - SYME constraints (should persist once added)
      - ELIM (track additions/removals)

   b. **Content comparison:** Beyond count, verify content matches:
      - Every DEFF NOT line in iteration N-1 must appear in iteration N
      - Every BOND line in iteration N-1 must appear in iteration N (unless advisory says remove)
      - Every PROP/LIST/ELEM grouping in iteration N-1 must appear in iteration N
      - Parenthesized HMBC syntax (e.g., `HMBC (5 6) 10`) must not revert to simple syntax

   c. **Expected changes:** Only these changes are normal:
      - New HMBC correlations added (incremental strategy)
      - ELIM added (last resort, should have documented reasoning)
      - SYME added (from grouping detection)
      - Constraints removed per explicit advisory from orchestrator

   **v3.0 UAT Bug Checklist (~35 lines — NEW content, from MEMORY.md):**

   These 5 specific bugs were found in live ibuprofen UAT (v3.0). Check for ALL of them on EVERY validation:

   1. **DEFF NOT patterns dropped** (CRITICAL severity):
      - Bug: Agent wrote 6 DEFF NOT patterns in iteration 1, then forgot to carry them forward to iterations 2-4
      - Check: Count DEFF NOT lines in current file. If iteration > 1 and count is less than iteration 1, flag CRITICAL
      - Expected patterns (for natural products): C1CC1, C1CCC1, C1NC1, C1NCC1, C1SC1, C1SCC1, C1OC1, C1OCC1

   2. **Signal grouping detected but never applied** (WARNING severity):
      - Bug: Grouping detection found close signals but agent never translated to SYME or parenthesized HMBC
      - Check: If nmr-chemist reported signal groups, verify either SYME constraint or parenthesized HMBC syntax exists
      - Cross-reference: Read nmr-chemist's detection results message

   3. **Grouped atom notation dropped** (CRITICAL severity):
      - Bug: Iteration 1 used `HMBC (6 7) 2` but iterations 2-4 reverted to `HMBC 6 2`
      - Check: If iteration N-1 has parenthesized HMBC, iteration N must preserve them
      - Regex check: Compare `HMBC \(` occurrences between files

   4. **PROP not used despite being documented** (INFO severity):
      - Bug: CASE-PROGRESS said "PROP constraint for O neighbor" but LSD file used BOND instead
      - Check: If CASE-PROGRESS.md mentions PROP, verify LSD file actually contains PROP (or document why BOND was used instead)
      - Note: Using BOND instead of PROP is valid if justified — flag as INFO not CRITICAL

   5. **No ELIM/LIST constraints from detection** (WARNING severity):
      - Bug: Neighbourhood detection ran but only influenced BOND C=O; no flexible constraints written
      - Check: If nmr-chemist reported mandatory/forbidden neighbors beyond carbonyl, verify corresponding PROP/LIST constraints exist
      - Cross-reference: Read nmr-chemist's neighbourhood detection results

   **Structural Integrity Checks (~30 lines — from monolith Section 3):**

   These checks apply to EVERY LSD file regardless of iteration:

   a. **sp2 count parity**: Count all atoms with hybridization 2. Must be EVEN. If odd, flag CRITICAL with specific atom list.

   b. **H budget balance**: Sum all hydrogen counts from MULT definitions (including heteroatom H). Compare against molecular formula hydrogen count. If mismatch, flag CRITICAL with counts.

   c. **Correlation order**: Verify all HSQC commands appear before any HMBC commands. If violated, flag CRITICAL.

   d. **HMBC reference validity**: Every HMBC correlation must reference an atom defined by HSQC or MULT. If referencing undefined atom, flag CRITICAL.

   e. **Badlist completeness**: For natural products, expect at minimum: C1CC1, C1CCC1 DEFF NOT patterns. If missing, flag WARNING.

   f. **ELIM usage**: If ELIM is present in iteration 1, flag WARNING ("ELIM should be last resort, not first run"). If ELIM N > 3, flag WARNING ("Excessive elimination suggests other issues").

   **Severity Classification (~15 lines):**

   | Severity | Meaning | Action Required |
   |----------|---------|-----------------|
   | CRITICAL | Solver will fail or produce wrong results | BLOCK solver run. LSD-Engineer must fix before proceeding. |
   | WARNING | Potential constraint loss or suboptimal strategy | Flag to team. LSD-Engineer should address but can proceed with justification. |
   | INFO | Observation or minor discrepancy | Log for CASE-PROGRESS.md. No blocking action. |

4. `<message_interface>` section (~12 lines):

   **OUTPUTS:**
   - Validation report: list of issues with severity (CRITICAL/WARNING/INFO)
   - Approval or rejection: "APPROVED — no critical issues" or "BLOCKED — {N} critical issues found"
   - Constraint diff summary: what changed between iterations

   **INPUTS:**
   - From lsd-engineer: "ready for validation" notification with LSD file path and iteration number
   - From nmr-chemist: detection results (for cross-referencing grouping/detection checks)
   - From orchestrator: advisory constraints, task assignments

5. `<workflow>` section (~15 lines):
   - Step 1: Receive "ready for validation" from lsd-engineer
   - Step 2: Read current LSD file
   - Step 3: If iteration > 1: Read previous iteration's LSD file for diff
   - Step 4: Run structural integrity checks (sp2, H budget, correlation order)
   - Step 5: Run constraint diff (count and content comparison)
   - Step 6: Run v3.0 UAT bug checklist (all 5 items)
   - Step 7: Classify each issue by severity
   - Step 8: If any CRITICAL: send "BLOCKED" with issue list to lsd-engineer
   - Step 9: If no CRITICAL: send "APPROVED" (with WARNINGs/INFOs listed) to lsd-engineer
   - Step 10: Post validation summary to team via SendMessage
