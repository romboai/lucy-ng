4. Add migration function `migrate_v3_to_v4(conn: sqlite3.Connection) -> None`:
   - Uses ALTER TABLE ADD COLUMN for each of the 3 new columns with DEFAULT 0
   - Creates the idx_hose_stats_mean_radius index
   - Updates schema_meta to version 4
   - Commits the transaction
   Note: SQLite ALTER TABLE ADD COLUMN is safe and fast -- it only modifies the schema, not existing rows.

In manager.py:
1. Add method `migrate_to_v4(self) -> bool`:
   - Calls `get_schema_version()` to check current version
   - If version < 4, calls `migrate_v3_to_v4(self.connection)` from schema.py
   - Returns True if migration was performed, False if already at v4+
   - Logs via click.echo or returns silently (no logging dependency needed)

2. Add method `get_hose_stats_by_shift_window(self, shift_ppm: float, radius: int, window_ppm: float = 2.0, min_count: int = 5) -> list[HOSEStatsRecord]`:
   - Queries hose_stats WHERE radius = ? AND mean BETWEEN (shift - window) AND (shift + window) AND count >= min_count
   - SELECT includes all columns: hose_code, radius, mean, std, count, sp3_count, sp2_count, sp1_count
   - Returns list of HOSEStatsRecord constructed from rows
   - If sp3_count/sp2_count/sp1_count columns don't exist (v3 database), catch sqlite3.OperationalError and fall back to query without those columns (returning records with 0 for detection fields)
   - This backward compatibility is important: users may have v3 databases that haven't been migrated yet

3. Update `upsert_hose_stats_incremental` to handle the 3 hybridisation columns:
   - Extend the input tuple format to include sp3_count, sp2_count, sp1_count (optional, default 0)
   - For new inserts: include hybridisation columns
   - For merges (existing entries): add the new counts to old counts (simple addition, not Welford -- these are discrete counts not continuous values)
   - Accept tuples of length 5 (backward compat: hose_code, radius, count, mean, m2) OR length 8 (hose_code, radius, count, mean, m2, sp3_count, sp2_count, sp1_count)

4. Update existing `get_hose_stats` and `get_hose_stats_all_radii` methods to include the new columns in SELECT and HOSEStatsRecord construction. Use try/except for backward compatibility with v3 databases where columns don't exist.

   - test_get_hose_stats_by_shift_window: Create v4 database, insert hose_stats rows with known mean values (e.g., 25.0, 130.0, 155.0), query with shift=130.0 window=2.0, verify only the 130.0 row is returned
   - test_shift_window_min_count: Insert rows with count=3 and count=10, verify min_count=5 filters correctly
   - test_upsert_with_hybridisation: Insert stats with sp3/sp2/sp1 counts, verify they are stored and retrieved correctly
