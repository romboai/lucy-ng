The document structure should be:

**1. Overview and Role** (~20 lines)
- Supervisor is the single entry point for ALL lucy-ng invocations
- Routes to dereplication, CASE, sanitize based on user intent
- For CASE: spawns CASE agent via Task tool, monitors via CASE-PROGRESS.md, intervenes when stuck

- Decision tree for routing (keep and expand existing one)
- Supervisor routes: sanitize (bash CLI), dereplication (bash CLI), full CASE (spawn CASE agent via Task tool)
- Default behavior: dereplication first, then CASE if no match

**3. CASE Workflow Supervision** (~40 lines)
- Supervisor spawns CASE agent with compound path, formula, and instructions to write CASE-PROGRESS.md
- After CASE agent completes an iteration batch (or returns), supervisor reads CASE-PROGRESS.md
- Supervisor checks for loop patterns (Section 4)
- If loop detected: diagnose from progress log, advise CASE agent with specific constraints, track intervention count
- Advisory intervention model: tell CASE agent WHAT the problem is and WHAT to fix, but NOT exactly HOW (CASE agent retains autonomy)
- Example advisory: "sp2 count issue detected -- verify sp2 count is even before retrying" (specific), NOT "try again" (generic)

**4. Loop Detection Patterns** (~80 lines)
Four loop patterns, each with: Definition, Detection criteria, Diagnostic procedure, Example advisory message.

  4.1 ELIM Thrashing (SUPV-02):
  - Definition: Adding ELIM repeatedly without diagnosing root cause
  - Detection: CASE-PROGRESS.md shows ELIM added in 2+ iterations, OR ELIM added then removed then added again
  - Diagnostic procedure: Check (1) sp2 atom count is even, (2) hydrogen budget matches formula, (3) HMBC correlations checked for 1J artifacts, (4) close carbons causing ambiguous assignment
  - Advisory: "ELIM thrashing detected. Before retrying: (1) verify sp2 count is even, (2) verify H budget matches formula, (3) check last batch of HMBC correlations for 1J artifacts (compare against HSQC positions). Do NOT add ELIM again until all three checks pass."

  4.2 Zero-Solution Loop (SUPV-03):
  - Definition: 3+ iterations returning 0 solutions without changing approach
  - Detection: CASE-PROGRESS.md shows 3+ consecutive iterations with solution_count = 0 AND constraints_added/removed follow same pattern
  - Diagnostic procedure: Check (1) sp2 count even, (2) H budget, (3) remove last batch and confirm solutions return, (4) test individual correlations from failed batch, (5) check molecular formula correctness
  - Advisory: "Zero-solution loop detected (3 consecutive 0-solution results). Diagnose: remove last HMBC batch, confirm solutions return. Then test each correlation individually to find the conflict. Check if any carbons within 3 ppm could be misassigned."

  4.3 Solution Explosion (SUPV-04):
  - Definition: 3+ iterations with solution count > 100 and < 10% reduction each
  - Detection: CASE-PROGRESS.md shows 3+ recent iterations all with solution_count > 100 AND each iteration reduces by less than 10%
  - Diagnostic procedure: Check (1) is ELIM present (remove it), (2) are added correlations actually constraining (check if they involve already-connected atoms), (3) are quaternary carbons unconstrained, (4) are heteroatom constraints missing
  - Advisory: "Solution explosion stalled (3 iterations, <10% reduction each, still >100 solutions). Check: (1) remove ELIM if present, (2) verify recent HMBC correlations connect NEW fragments (not already-connected atoms), (3) add heteroatom constraints (BOND or LIST/PROP for O, N atoms), (4) check quaternary carbons -- add shift-based constraints if 0 HMBC correlations."

  4.4 Constraint Churning (SUPV-05):
  - Definition: Adding and removing constraints randomly without convergence over 5+ iterations
  - Detection: CASE-PROGRESS.md shows 5+ recent iterations with high add/remove activity (>10 added, >5 removed) AND solution count not converging (still > 50)
  - Diagnostic procedure: Check (1) is a systematic strategy being followed (incremental HMBC per skill/SKILL.md Section 7), (2) are correlations being selected by confidence criteria or randomly, (3) is the molecular formula correct
  - Advisory: "Constraint churning detected (high add/remove activity without convergence). Reset to last known-good state (iteration with lowest non-zero solution count). Follow incremental HMBC strategy: select 3-5 HIGH-CONFIDENCE correlations (isolated carbon shifts, unique proton assignments, strong intensity). Do not add/remove randomly."

**5. Convergence Criteria** (~40 lines)
- Three convergence signals: solution count trend (should decrease), constraint effectiveness (each batch should change solution set), confidence score improvement
- Flexible success targets:
  - Ideal: 1-5 solutions with top-ranked high confidence
  - Acceptable: <10 solutions with top-ranked well-differentiated by ranking score
  - Not a failure if count slightly higher, as long as ranking clearly separates best candidates
- Hard safety cap: ~20 total LSD iterations maximum, then report best result and escalate
- Plateau handling: conditional
  - Plateau at <=10 solutions with good ranking differentiation = convergence (STOP, rank)
  - Plateau at >10 solutions = try additional strategy (heteroatom constraints, symmetry, different HMBC batch)

**6. Intervention Tracking and Escalation** (~30 lines)
- Track intervention count PER PATTERN (not global)
- Each intervention cycle: detect loop -> diagnose from progress log -> advise CASE agent with constraints -> CASE retries
- Escalate to user after 10 failed intervention cycles with SAME pattern (per CONTEXT.md decision, overrides SUPV-07 wording of "3 failed attempts")
- Escalation report format: what pattern was detected, how many times, what diagnostics were tried, what the supervisor recommends the user investigate
- Also escalate for: conflicting data between experiments, unusual chemical shifts outside normal ranges, molecular formula mismatch with observed data

**7. CASE-PROGRESS.md Format Specification** (~60 lines)
Full specification of the progress log format. Include:
- Header section: compound path, formula, started timestamp, CASE agent identifier
- Iteration section template with ALL required fields:
  - Iteration number and description (e.g., "Iteration 3: Add quaternary carbon HMBC batch")
  - Time
  - LSD file reference
  - Solution count
  - Constraints added (list each with reasoning)
  - Constraints removed (list each with reasoning)
  - Why: natural language explanation of what was done and why
  - Constraint effectiveness: % reduction or "over-constrained" or "baseline"
  - Confidence: qualitative assessment of current state
  - HMBC correlations used: X/Y (running total)
  - Notes: sp2 check, H budget check, other observations
- Append-only rule: each iteration APPENDS a new section; NEVER overwrite previous iterations
- Example: include a complete 3-iteration example showing baseline, productive batch, and failure recovery
- Reference the research document example (24-RESEARCH.md) for format inspiration

### Step 7c: Write Progress Checkpoint (CASE-PROGRESS.md)

## Iteration N: <brief description>

**Time:** <timestamp>
**LSD file:** <filename>.lsd
**Solution count:** <count>

- <constraint and reasoning>

**Constraints removed:**
- <constraint and reasoning> (or "None")

**Why:** <natural language explanation of strategy for this iteration>

**Constraint effectiveness:** <% reduction from previous, or "baseline", or "over-constrained (0 solutions)">
**Confidence:** <qualitative assessment: too many solutions / converging / stuck / etc.>
**HMBC correlations used:** X/Y

**Notes:**
- sp2 count: <N> (<even/odd>) <check/warning>
- H budget: <matches/mismatch>
- <other observations>
