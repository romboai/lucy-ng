Add a new section `## 5. Constraint Inventory System` inside the existing `<domain_knowledge>` block (after section 4). This section contains:

**A. JSON Schema Reference**

Document the inventory JSON schema from 43-RESEARCH.md. Include the complete field list with types and purpose:
- `version` (always 1), `iteration` (1-based), `formula`, `timestamp`
- `mult_count`, `hsqc_count`
- `hmbc_batches` (array of {batch, count, correlations}), `hmbc_total`
- `grouped_hmbc` (parenthesized HMBC syntax entries)
- `bond_constraints`, `syme_pairs`, `list_prop_constraints`, `elim_value`
- `deff_not_patterns` (SMARTS strings for DEFF NOT commands)
- `detection_results` (hybridisation, neighbours, hhb, grouping)
- `applied_from_detection`, `pending_from_detection`

**B. LSD File Format**

Show the inventory block format with delimiters:
```
; === CONSTRAINT INVENTORY v1 ===
; { JSON lines prefixed with "; " }
; === END CONSTRAINT INVENTORY ===
```

The inventory block goes at the TOP of the LSD file, BEFORE any MULT definitions. Each JSON line is prefixed with `; ` (LSD comment syntax). Include a compact example showing a realistic iteration-2 inventory.

**C. Initialization Procedure (Iteration 1)**

Step-by-step procedure for creating the first inventory:
1. After writing all MULT, HSQC, first HMBC batch, BOND, DEFF NOT to the LSD file
2. Count each constraint type from what was just written
3. Build the inventory JSON with all fields populated
4. Populate `deff_not_patterns` from the standard badlist set (these are constant)
5. Populate `detection_results` from nmr-chemist's detection message
6. Classify each detection result as `applied_from_detection` or `pending_from_detection`
7. Write the inventory block at the TOP of the file (before MULT lines)

Mark as CRITICAL: "The DEFF NOT patterns array MUST be populated at initialization. This is the primary defense against Bug 1."

**D. Update Procedure (Iteration N > 1)**

Step-by-step procedure for updating an existing inventory:
1. Read previous iteration's LSD file (already required by existing rule)
2. Extract the inventory JSON from between the `; === CONSTRAINT INVENTORY v1 ===` and `; === END CONSTRAINT INVENTORY ===` delimiters
3. Copy ALL fields verbatim from previous inventory as starting point
4. Update only these fields:
   - Increment `iteration`
   - Append new HMBC batch to `hmbc_batches`, increment `hmbc_total`
   - Update `timestamp`
   - If advisory says add/remove constraints: update relevant fields
   - Review `pending_from_detection`: apply any pending items that should now become constraints (move to `applied_from_detection`)
5. Write the updated inventory block at the TOP of the new LSD file
6. Write ALL LSD commands below (copied from previous + new batch)

Mark as CRITICAL: "Copy ALL inventory fields from previous iteration. NEVER rebuild the inventory from scratch — this is the same 'read previous, never reconstruct' rule applied to the inventory itself."

**E. Atomic Write Rule**

Add a boxed rule: "The inventory and the LSD commands are written as a SINGLE Write operation. The inventory counts MUST match the actual commands in the same file. Never write the inventory separately from the commands."

**Formatting guidelines:**
- Keep the new section concise (~60-80 lines). Use the same formatting style as existing sections (code blocks for examples, tables for reference, bold for critical rules).
- Do NOT duplicate the full JSON schema from research — summarize the fields in a compact table and show one realistic example.
- Keep arrays compact on single lines where possible.

**Format commitment:** The inventory format is JSON embedded in `;`-prefixed comments as defined in 43-RESEARCH.md Pattern 1. Do NOT use the key-value fallback (Pitfall 4 mitigation format). The LSD parser compatibility test in this plan's verify section confirms JSON characters in `;` comments are safe.
